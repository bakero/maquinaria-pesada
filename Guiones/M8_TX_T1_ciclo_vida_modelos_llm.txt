# BLOQUE_QUE

IAGO: [didactico] LLMOps es una especialización de MLOps adaptada a las características únicas de los modelos de lenguaje. No es simplemente poner un modelo en un servidor y esperar que funcione. Es el conjunto de prácticas, herramientas y procesos que hacen que un sistema LLM sea fiable, escalable y mejorable en producción. Cuatro características hacen que los LLMs sean distintos a otros sistemas de machine learning: comportamiento no determinista, outputs difíciles de evaluar automáticamente, coste basado en tokens, y vulnerabilidades específicas como prompt injection o alucinaciones.

MARIA: ¿Por qué no determinista?

IAGO: Porque el mismo input puede producir outputs distintos. Si le preguntas tres veces lo mismo a GPT-4 con temperatura mayor que cero, obtendrás tres respuestas distintas. Eso complica el testing, la reproducibilidad y el debugging. En un sistema tradicional de software, la misma entrada produce siempre la misma salida. Aquí no.

MARIA: [esceptico] Y eso de que los outputs son difíciles de evaluar, ¿qué significa en la práctica?

IAGO: Que no puedes medir la calidad del texto con una sola métrica numérica. En clasificación de imágenes tienes accuracy o F1-score. En un chatbot de soporte técnico, ¿cómo mides si la respuesta fue útil? ¿Si resolvió el problema del usuario? ¿Si el tono fue apropiado? Necesitas métricas semánticas, evaluación humana, o usar otro LLM como juez. Eso hace que la evaluación sea más cara y más lenta que en otros sistemas de ML.

IAGO: [explicativo] El coste basado en tokens significa que cada consulta tiene un coste variable. No pagas por el tiempo de cómputo, pagas por la cantidad de texto procesado. Si un usuario hace una pregunta de 50 tokens y el modelo responde con 500 tokens, esa consulta cuesta diez veces más que una de 50 tokens de entrada y 50 de salida. Eso obliga a monitorizar el coste por consulta en tiempo real y a optimizar el diseño del prompt para minimizar tokens sin sacrificar calidad.

MARIA: ¿Y las vulnerabilidades específicas?

IAGO: Prompt injection: un usuario malicioso puede intentar manipular el comportamiento del sistema inyectando instrucciones en su input. Alucinaciones: el modelo genera información falsa con total confianza. Sesgos: el modelo reproduce sesgos presentes en los datos de entrenamiento. Ninguna de estas vulnerabilidades existe en un sistema tradicional de software. Requieren técnicas de mitigación específicas: validación de inputs, fact-checking, evaluación de fairness.

# BLOQUE_COMO

IAGO: [explicativo] El ciclo de vida de un sistema LLM tiene cuatro fases. Primera fase: diseño y selección del modelo. Defines los requisitos del sistema: qué tareas debe resolver, qué latencia es aceptable, cuál es el presupuesto de coste por consulta, cuáles son los requisitos de privacidad y compliance, y qué volumen de consultas esperas. Con esos requisitos, seleccionas el modelo base: modelo frontier via API como GPT-4 o Claude, o modelo open source autohosteado como Llama o Mistral. Decides el enfoque de adaptación: prompting directo, RAG, fine-tuning, o una combinación. Y eliges el stack tecnológico: qué framework de orquestación, qué base de datos vectorial, qué plataforma de observabilidad.

MARIA: [conversacional] Segunda fase: desarrollo y evaluación. Aquí construyes el sistema y estableces la línea base de rendimiento. Diseñas el prompt o el pipeline RAG. Construyes el golden dataset: entre 50 y 500 casos representativos del problema que quieres resolver, con inputs reales y outputs esperados. Si haces fine-tuning, preparas el dataset de entrenamiento. Y evalúas rigurosamente el sistema sobre el golden dataset con métricas apropiadas para tu caso de uso. Esta línea base documentada es crítica: es la referencia que te permitirá detectar regresiones cuando despliegues cambios en producción.

MARIA: [directo] Tercera fase: despliegue. Configuras el servidor de inferencia: vLLM, Text Generation Inference, o la API de tu proveedor. Configuras el entorno de producción: variables de entorno, secretos, rate limiting, políticas de retry. Y haces un despliegue gradual: canary deployment al 5% del tráfico durante una semana, o shadow mode donde el nuevo sistema procesa las consultas pero no devuelve respuestas a los usuarios todavía. Verificas que el sistema funciona correctamente bajo condiciones reales antes de abrir completamente.

IAGO: [serio] Cuarta fase: monitorización y mejora continua. Esta es la fase más larga y la más crítica. Haces logging de todas las interacciones: inputs, outputs, latencia, coste. Evalúas continuamente la calidad: muestreas el 5-10% de las respuestas y las evalúas con métricas automáticas o LLM-as-judge. Detectas drift: cambios en la distribución de inputs o degradación de la calidad. Configuras alertas basadas en umbrales de SLO. Y ejecutas el ciclo de mejora: identificas los casos de peor rendimiento, iteras el prompt o el sistema, evalúas el impacto, y redespliegas.

MARIA: ¿Qué son los SLOs?

IAGO: Service Level Objectives. Los compromisos de calidad del sistema. Para un sistema LLM típico: latencia P99 menor de 3 segundos, disponibilidad mayor del 99.5%, tasa de errores menor del 0.1%, faithfulness mayor de 0.85 para sistemas RAG, y tasa de alucinaciones menor del 5%. Cuando un SLO se viola, se dispara una alerta y el ingeniero de guardia investiga y actúa. Sin SLOs, no tienes forma objetiva de saber si tu sistema está funcionando bien o mal.

# BLOQUE_LIMITES

MARIA: [esceptico] Ahora hablemos de lo que falla. Primer error: no versionar los prompts junto con el código. He visto equipos que tratan los prompts como configuración temporal, los editan directamente en la interfaz de la plataforma, y no los commitean a Git. Un cambio de prompt es un cambio de sistema que puede mejorar o degradar el rendimiento en un 20%. Sin versionado con hash de commit, es imposible reproducir el estado exacto del sistema en cualquier momento histórico o hacer rollback a una versión específica del prompt. Si algo se rompe en producción y no sabes qué versión del prompt estaba corriendo, estás perdido.

IAGO: [reflexivo] Segundo error: monitorizar solo la latencia y el coste, no la calidad. Los dashboards técnicos de latencia P99 y disponibilidad son necesarios, pero no capturan si el sistema está produciendo respuestas útiles o correctas. La calidad del output debe monitorizarse con métricas semánticas: relevancia, coherencia, faithfulness en sistemas RAG. Sin monitorización de calidad, el sistema puede degradarse silenciosamente durante semanas antes de que los usuarios se quejen. Y cuando se quejan, ya perdiste su confianza.

MARIA: [contundente] Tercer error: despliegue big-bang en lugar de gradual. Desplegar un cambio al 100% del tráfico inmediatamente elimina la capacidad de detectar problemas antes de que afecten a todos los usuarios. Un caso real: una empresa fintech actualizó su modelo base de GPT-4o a GPT-4o-2025-04 sin canary deployment. La actualización provocó una degradación del 8% en la métrica de faithfulness de su sistema RAG financiero. El sistema de monitorización detectó la degradación en 4 horas. Hicieron rollback al modelo anterior en 15 minutos. El análisis post-mortem reveló que el nuevo modelo seguía las instrucciones de formato de forma distinta. Tuvieron que actualizar el prompt antes de re-desplegar el nuevo modelo. Si hubieran hecho canary deployment al 5% del tráfico, habrían detectado el problema afectando solo a 5% de los usuarios, no al 100%.

IAGO: Cuarto error: no tener un proceso de rollback documentado y probado. En producción, las cosas se rompen. Un cambio de prompt que funcionaba perfectamente en el golden dataset puede fallar con inputs reales que no habías anticipado. Necesitas poder volver a la versión anterior en minutos, no en horas. Eso requiere versionado de prompts, versionado de modelos, y un proceso claro de quién tiene autoridad para ejecutar el rollback y bajo qué condiciones.

MARIA: [ironico] Y el quinto error, el más sutil: no construir el golden dataset con suficiente cuidado. He visto equipos que construyen el golden dataset en una tarde con 20 ejemplos inventados por el ingeniero. Ese dataset no representa la distribución real de inputs que el sistema verá en producción. Cuando despliegas, descubres que el 30% de las consultas reales son de un tipo que no habías incluido en el golden dataset. Tu línea base de evaluación no sirve. Construir un buen golden dataset requiere tiempo: recoger inputs reales de usuarios, curar manualmente los outputs esperados, cubrir los edge cases, y actualizar el dataset continuamente conforme el sistema evoluciona.

MARIA: Primero: LLMOps es MLOps adaptado a las características únicas de los LLMs: comportamiento no determinista, métricas semánticas, prompts como código, y vulnerabilidades específicas.
