# BLOQUE_PANORAMA

IAGO: [didactico] RAG es el acrónimo de Retrieval-Augmented Generation. Generación aumentada con recuperación. Es una arquitectura que resuelve el problema más básico de los modelos de lenguaje en empresa: el modelo sabe del mundo hasta su fecha de corte de entrenamiento, pero no sabe nada de tus documentos internos, de tus políticas actualizadas, de los datos de tus clientes. RAG soluciona esto sin modificar el modelo. En lugar de reentrenar, le das al modelo la información relevante directamente en el contexto de cada consulta.

MARIA: [curioso] ¿Cómo un libro abierto en lugar de un examen de memoria?

IAGO: [claro] ese símil. Un modelo sin RAG responde de memoria cerrada: solo puede usar lo que aprendió en el preentrenamiento. Con todas sus limitaciones. Corte de conocimiento, posibles alucinaciones, cero acceso a información privada. Un modelo con RAG responde con libro abierto: antes de generar la respuesta, consulta las fuentes relevantes y construye su respuesta basándose en ellas. Por eso RAG se ha convertido en el caso de uso empresarial número uno. El mercado alcanzó mil ochocientos cincuenta millones de dólares en dos mil veinticuatro y crece al cuarenta y nueve por ciento anual. Anthropic, OpenAI y Google lo recomiendan como técnica principal para conectar modelos con datos propietarios.

MARIA: [esceptico] ¿Y por qué no hacer fine-tuning directamente?

IAGO: [firme] Porque RAG y fine-tuning responden a necesidades diferentes. RAG es la elección correcta cuando el conocimiento cambia frecuentemente. Documentos que se actualizan, datos en tiempo real. Cuando la base de conocimiento es grande y diversa. Cuando necesitas trazabilidad: de qué documento viene esta respuesta. O cuando los datos son confidenciales y no quieres incluirlos en el proceso de entrenamiento. Fine-tuning es mejor cuando necesitas cambiar el estilo, el tono o el comportamiento del modelo. Cuando el dominio es estable y bien definido. O cuando el rendimiento de RAG no es suficiente para la tarea específica.

# BLOQUE_COMO

IAGO: [explicativo] El flujo básico de RAG tiene cuatro etapas. Primera: ingesta. Los documentos de la base de conocimiento se dividen en fragmentos manejables, lo que llamamos chunks. Se generan sus embeddings, sus representaciones vectoriales. Y se almacenan en una base de datos vectorial junto con sus metadatos. Segunda: recuperación. Cuando llega una consulta del usuario, se genera su embedding y se buscan los K chunks más similares en la base vectorial. Tercera: aumentación. Los chunks recuperados se insertan en el prompt del modelo como contexto adicional. Cuarta: generación. El modelo genera una respuesta basándose en la consulta y el contexto recuperado.

MARIA: [analitica] Vamos a los componentes técnicos. ¿Qué necesitas para montar un sistema RAG en producción?

IAGO: [reflexivo] Un sistema RAG completo tiene cinco componentes. El pipeline de ingesta procesa los documentos fuente: parsing, que es extracción de texto de PDF, Word, HTML. Chunking, división en fragmentos. Embedding, vectorización de cada chunk. Y almacenamiento en la base vectorial con metadatos.

MARIA: [directo] El segundo componente es la base de datos vectorial. Almacena embeddings e indexa para búsqueda eficiente por similitud. En diciembre de dos mil veinticinco, el mercado se consolida alrededor de cuatro opciones. Pinecone: SaaS managed, más fácil de empezar. Weaviate: open source más cloud, buen balance. Milvus: open source, mayor escala. Y Qdrant: open source, alto rendimiento, fácil de auto-hostear. FAISS de Meta es la librería estándar para búsqueda vectorial en memoria sin necesidad de una base de datos completa.

IAGO: [conversacional] El tercer componente es el sistema de retrieval. Procesa la consulta del usuario, genera su embedding, y recupera los chunks más relevantes. El cuarto es el prompt builder: construye el prompt del modelo combinando la consulta y el contexto recuperado. Y el quinto es el modelo generador: produce la respuesta fundamentada en el contexto.

MARIA: [curioso] ¿Y cómo decides el tamaño del chunk?

IAGO: [pausado] Esa es una de las decisiones de diseño más críticas. Chunks muy pequeños pierden contexto. Chunks muy grandes diluyen la señal. No hay una respuesta única. Depende del tipo de preguntas que el sistema debe responder. Por eso el proceso correcto empieza definiendo los veinte a cincuenta casos de uso más representativos. Construyes el golden dataset de evaluación. Y diseñas el sistema para maximizar el rendimiento en esos casos.

Usa un modelo de lenguaje con fine-tuning sobre corpus jurídico, integrado con bases de datos legales como Westlaw y LexisNexis. La adopción: más del noventa y siete por ciento de los grandes bufetes de Estados Unidos en lista de espera durante dos mil veintitrés. Casos de uso: redacción de contratos, due diligence, investigación jurídica, análisis de precedentes. MARIA: [contundente] Morgan Stanley. Sistema RAG para asesores financieros. Base de conocimiento con más de cien mil documentos internos: informes de análisis, presentaciones de producto, políticas de compliance. Los asesores hacen preguntas en lenguaje natural y el sistema recupera los documentos relevantes y genera un resumen contextualizado. Resultado: reducción del tiempo de búsqueda de información de horas a minutos. Socio tecnológico: OpenAI con GPT-cuatro. Presentación pública en OpenAI DevDay dos mil veintitrés.

MARIA: [directo] Otro caso: JPMorgan COIN. Contract Intelligence. Basado en machine learning clásico: procesamiento de lenguaje natural, clasificadores. Analiza contratos legales de préstamos comerciales y extrae cláusulas clave. Resultado cuantificado: trescientas sesenta mil horas de trabajo de abogados y analistas ahorradas al año. Antes era un proceso manual que llevaba trescientas sesenta mil horas anuales. Ahora tarda segundos. Es el caso canónico de machine learning clásico con ROI cuantificado y verificable.

IAGO: [reflexivo] ¿Cuál es el reto más grande?

MARIA: [analitica] El gap entre prototipo y producción. Construir un RAG que funciona en un notebook es fácil. Llevarlo a producción con gestión del corpus, latencia controlada, multi-tenancy, y monitorización continua es donde la mayoría de los proyectos se quedan atascados. Requiere definir criterios de calidad del corpus, actualización de documentos, evaluación continua con métricas como RAGAS, y arquitectura que escale. El stack tecnológico de referencia en dos mil veinticinco: LangChain o LlamaIndex como framework de orquestación, Qdrant o Pinecone como base vectorial, Voyage AI o text-embedding-tres-large como modelo de embedding, un modelo frontier o modelo fine-tuneado como generador, y RAGAS para evaluación continua.

MARIA: [calido] Y un error común: asumir que RAG elimina totalmente las alucinaciones. RAG reduce dramáticamente las alucinaciones de conocimiento, el modelo afirma cosas que no están en los documentos, pero no las elimina completamente. El modelo puede aún distorsionar o mal interpretar el contexto recuperado. La evaluación continua con métricas de faithfulness es esencial.
