# BLOQUE_PANORAMA

IAGO: [didactico] RAG significa Retrieval-Augmented Generation. Es una arquitectura que combina dos capacidades: recuperar información de bases de conocimiento propias y generar respuestas con modelos de lenguaje. Piensa en la diferencia entre un examen de memoria cerrado y un examen con libro abierto. Un modelo sin RAG responde de memoria: solo usa lo que aprendió durante el entrenamiento, con todas sus limitaciones. Corte de conocimiento, posibles alucinaciones, cero acceso a información privada. Un modelo con RAG responde con libro abierto: antes de generar la respuesta, consulta las fuentes relevantes y construye su respuesta basándose en ellas.

MARIA: [curioso] ¿Y por qué es el caso de uso número uno?

IAGO: [firme] Porque resuelve el problema fundamental de conectar modelos generales con datos propietarios empresariales. Los modelos tienen conocimiento del mundo hasta su fecha de corte, pero no saben nada sobre tus documentos internos, tus políticas actualizadas, tus datos de clientes. RAG soluciona esto sin modificar el modelo. En lugar de hacer fine-tuning, le proporcionas la información relevante directamente en el contexto de cada consulta. El mercado alcanzó mil ochocientos cincuenta millones de dólares en dos mil veinticuatro y crece al cuarenta y nueve por ciento anual. Anthropic, OpenAI y Google recomiendan RAG como técnica principal para datos propietarios.

MARIA: [esceptico] ¿Cuándo usas RAG en lugar de fine-tuning?

IAGO: [explicativo] RAG es la elección correcta cuando el conocimiento cambia frecuentemente. Documentos que se actualizan, datos en tiempo real, políticas que evolucionan. También cuando necesitas trazabilidad: saber de qué documento viene cada respuesta. O cuando los datos son confidenciales y no quieres incluirlos en el proceso de entrenamiento. Fine-tuning es mejor cuando necesitas cambiar el estilo, el tono o el comportamiento del modelo de forma permanente. Cuando el dominio es estable y bien definido. O cuando el rendimiento de RAG no es suficiente para la tarea específica.

# BLOQUE_COMO

IAGO: [didactico] El flujo básico de RAG tiene cuatro etapas. Primera: ingesta. Los documentos de la base de conocimiento se dividen en fragmentos manejables, llamamos a esto chunking. Cada fragmento se convierte en un embedding, un vector numérico que representa su significado. Esos embeddings se almacenan en una base de datos vectorial junto con sus metadatos: título del documento, fecha, sección, autor. Segunda etapa: recuperación. Cuando llega una consulta del usuario, se genera su embedding. Se buscan los K fragmentos más similares en la base vectorial usando distancia coseno o producto punto. Tercera: aumentación. Los fragmentos recuperados se insertan en el prompt del modelo como contexto adicional. Cuarta: generación. El modelo genera una respuesta basándose en la consulta original y el contexto recuperado.

MARIA: [analitica] ¿Qué componentes técnicos necesitas para construir esto?

IAGO: [explicativo] Un sistema RAG completo tiene cinco componentes. El pipeline de ingesta procesa los documentos fuente: parsing para extraer texto de PDF, Word, HTML. Chunking para dividir en fragmentos. Embedding para vectorizar cada fragmento. Y almacenamiento en la base vectorial con metadatos. Segundo componente: la base de datos vectorial. Almacena embeddings e indexa para búsqueda eficiente por similitud. Tercer componente: el sistema de retrieval. Procesa la consulta del usuario, genera su embedding, y recupera los fragmentos más relevantes. Cuarto: el prompt builder. Construye el prompt del modelo combinando la consulta y el contexto recuperado. Quinto: el modelo generador que produce la respuesta fundamentada en el contexto.

MARIA: [reflexivo] Hablemos de las bases vectoriales. Imagina que tienes que elegir una para tu empresa. ¿Qué opciones reales existen ahora mismo? Porque el mercado se ha consolidado bastante. Tienes cuatro opciones principales. Pinecone es la opción SaaS managed, la más fácil para empezar: te olvidas de la infraestructura. Weaviate es open source con versión cloud, buen balance entre control y comodidad. Milvus es open source para mayor escala, pensado para casos con millones de vectores. Y Qdrant, también open source, con alto rendimiento y fácil de auto-hostear si necesitas mantener los datos en tu infraestructura. Si no necesitas una base de datos completa, FAISS de Meta es la librería estándar para búsqueda vectorial en memoria.

IAGO: [conversacional] ¿Y qué pasa con los embeddings?

MARIA: [claro] Los embeddings son el componente central de la recuperación. La calidad del embedding determina qué tan bien el sistema encuentra los fragmentos relevantes. Los modelos de embedding genéricos funcionan bien para casos generales, pero tienen peor rendimiento en dominios especializados: médico, legal, técnico. Por eso evalúas el rendimiento de recuperación con tu golden dataset usando varios modelos antes de construir el índice completo. Text-embedding-three-large de OpenAI y Voyage AI son opciones estándar actualmente.

# BLOQUE_REALIDAD

MARIA: [firme] Vamos a los casos reales. Harvey AI es el ejemplo canónico en el sector legal. Startup fundada en dos mil veintidós, especializada en IA para bufetes. Usan un modelo de lenguaje con fine-tuning sobre corpus jurídico, integrado con bases de datos legales como Westlaw y LexisNexis. Más del noventa y siete por ciento de los grandes bufetes de Estados Unidos estaban en lista de espera durante dos mil veintitrés. El caso de uso: redacción de contratos, due diligence, investigación jurídica, análisis de precedentes. El resultado documentado: reducción de entre treinta y setenta por ciento del tiempo en tareas de revisión documental según testimonios de usuarios piloto.

IAGO: [conversacional] ¿Y en finanzas?

MARIA: [analitica] Morgan Stanley implementó un sistema RAG para asesores financieros. Base de conocimiento con más de cien mil documentos internos: informes de análisis, presentaciones de producto, políticas de compliance. Los asesores hacen preguntas en lenguaje natural y el sistema recupera los documentos relevantes y genera un resumen contextualizado. El resultado: reducción del tiempo de búsqueda de información de horas a minutos. El socio tecnológico es OpenAI con GPT-cuatro. Esto lo presentaron públicamente en OpenAI DevDay dos mil veintitrés.

MARIA: [esceptico] Microsoft tiene un chatbot de soporte técnico sobre la documentación completa de Azure. RAG sobre toda la documentación técnica. Los usuarios obtienen respuestas precisas a preguntas técnicas complejas con referencias a la sección exacta de la documentación. Esto reduce los tickets de soporte repetitivos. Otro caso: asistente de política interna de recursos humanos en una multinacional. RAG sobre el manual del empleado, convenios colectivos, políticas actualizadas. Los empleados consultan directamente sin necesidad de abrir tickets para preguntas sobre vacaciones, beneficios o procedimientos de baja.

IAGO: [reflexivo] ¿Cuál es el reto principal?

MARIA: [tenso] El gap entre prototipo y producción. Construir un prototipo de RAG es relativamente sencillo: LangChain, una base vectorial, un modelo de embedding, y un prompt básico. Pero llevarlo a producción requiere gestión del corpus: qué documentos incluir, con qué criterios de calidad y actualización. Requiere latencia controlada: los usuarios no esperan más de tres segundos. Requiere multi-tenancy: cada cliente ve solo sus documentos. Y requiere monitorización continua: evaluar si las respuestas son correctas, si los fragmentos recuperados son relevantes, si el sistema alucina. El stack tecnológico de referencia actualmente: LangChain o LlamaIndex como framework de orquestación, Qdrant o Pinecone como base vectorial, Voyage AI o text-embedding-three-large como modelo de embedding, un modelo frontier o fine-tuneado como generador, y RAGAS para evaluación continua.

MARIA: [directo] El error más común: construir el RAG antes de definir las preguntas que debe responder. El diseño del sistema depende del tipo de preguntas. Tamaño de chunk, modelo de embedding, parámetros de recuperación. Todo eso se optimiza para los casos de uso representativos. Empieza definiendo los veinte a cincuenta casos más representativos, construye el golden dataset de evaluación, y diseña el sistema para maximizar el rendimiento en esos casos. Otro error: asumir que RAG elimina totalmente las alucinaciones. RAG reduce dramáticamente las alucinaciones de conocimiento, el modelo afirma cosas que no están en los documentos. Pero no las elimina completamente. El modelo puede aún distorsionar o mal interpretar el contexto recuperado. La evaluación continua con métricas de faithfulness es esencial.
