# BLOQUE_COMO
IAGO: [didactico] Ahora vamos a bajar al mecanismo técnico. ¿Cómo funciona realmente la inteligencia artificial moderna? Todos los sistemas de IA, independientemente de su paradigma, comparten una estructura común. Aprenden patrones de datos mediante optimización matemática y los usan para hacer predicciones o generar salidas sobre entradas nuevas. La diferencia entre paradigmas está en cómo representan el conocimiento, qué tipo de datos procesan y qué tipo de salidas producen.
<break time="500ms"/>
MARIA: [explicativo] Vamos a desglosar eso en pasos concretos. Paso uno: recopilas datos relevantes para el problema. Pueden ser transacciones financieras, textos de contratos, imágenes médicas, señales de sensores industriales. Lo que sea relevante para tu caso de uso. Paso dos: usas esos datos para entrenar un modelo. Un modelo es una función matemática con millones o miles de millones de parámetros ajustables. El entrenamiento consiste en ajustar esos parámetros para que la función produzca salidas correctas en los ejemplos conocidos. Es un proceso de optimización iterativa.
<break time="500ms"/>
IAGO: [claro] Paso tres: el modelo entrenado se despliega en producción. Ahora produce predicciones o generaciones sobre datos nuevos que nunca ha visto durante el entrenamiento. Y paso cuatro: esas predicciones se integran en procesos de negocio para generar valor. Aprobas un préstamo, personalizas una oferta, detectas un fraude, automatizas una respuesta. Ese es el ciclo universal. Datos, modelo, predicción, valor.
<break time="500ms"/>
MARIA: [analitica] Ahora, los modelos de lenguaje de gran escala, los elemenes, añaden una capa de complejidad interesante. Son modelos entrenados sobre cantidades masivas de texto para predecir el siguiente token en una secuencia. Un token es básicamente una palabra o un fragmento de palabra. El modelo lee millones de documentos y aprende patrones estadísticos de qué palabras suelen seguir a qué otras palabras en qué contextos. Ese entrenamiento, repetido sobre billones de ejemplos, produce capacidades emergentes sorprendentes. Razonamiento, traducción, generación de código, síntesis de documentos.
<break time="500ms"/>
IAGO: [contundente] Pero aquí viene lo crítico. Esas capacidades son estadísticas, no comprensión. El modelo genera lo que es estadísticamente probable dado el contexto, no lo que es necesariamente verdad. No tiene acceso a hechos verificados. Tiene acceso a patrones de texto. Cuando le preguntas algo fuera de esos patrones, o algo que requiere razonamiento preciso que va más allá de la estadística lingüística, genera texto que parece correcto pero puede ser incorrecto. Eso es lo que llamamos alucinación.
<break time="500ms"/>
MARIA: [explicativo] Y las alucinaciones no son bugs que se puedan arreglar con un parche. Son consecuencias estructurales del paradigma. El modelo está diseñado para completar texto de forma coherente, no para verificar verdad. Entonces, ¿cómo gestionas esto en producción? La respuesta técnica más adoptada se llama rag: Retrieval Augmented Generation. Generación aumentada con recuperación. Funciona así: cuando el usuario hace una pregunta, el sistema primero busca en una base de documentos verificados los fragmentos más relevantes. Luego pasa esos fragmentos al elemen junto con la pregunta. El elemen genera la respuesta anclada a esos documentos. No inventa, sintetiza.
<break time="500ms"/>
IAGO: [claro] rag convierte un elemen genérico en un sistema especializado con conocimiento verificado. Es la arquitectura dominante en aplicaciones empresariales de IA generativa hoy. Y hay cuatro componentes técnicos que tienes que entender. Primero, la base de documentos. Segundo, el sistema de indexación y búsqueda semántica que recupera los fragmentos relevantes. Tercero, el promp que combina la pregunta del usuario con los documentos recuperados. Y cuarto, el elemen que genera la respuesta final.
<break time="500ms"/>
MARIA: [reflexivo] Cada uno de esos componentes puede fallar de formas distintas. Si la base de documentos está desactualizada, las respuestas serán correctas pero obsoletas. Si el sistema de búsqueda no encuentra los documentos relevantes, el elemen generará respuestas sin fundamento. Si el promp no está bien diseñado, el modelo puede ignorar los documentos y alucinar. Y si el elemen no es suficientemente potente, puede malinterpretar los documentos. Por eso rag no es una solución mágica. Es una arquitectura que requiere ingeniería cuidadosa en cada capa.

MARIA: [analitica] ¿Qué significa degradación en este contexto?
<break time="500ms"/>
IAGO: [explicativo] Significa que el modelo fue entrenado con datos de un periodo específico. Si las condiciones cambian, el modelo empieza a cometer más errores. Ejemplo: un modelo de detección de fraude entrenado con datos de dos mil veintitres puede no detectar patrones de fraude nuevos que aparecen en dos mil veinticinco. Necesitas reentrenar o ajustar el modelo regularmente. Eso requiere infraestructura, datos actualizados y equipos con capacidad técnica.
IAGO: [contundente] Segundo límite: la IA no debe confundirse con comprensión humana. Los elemenes generan texto estadísticamente probable. No tienen creencias, no tienen intenciones, no entienden en el sentido humano. Atribuirles estas propiedades lleva a confiar en ellos en situaciones donde no se debe. Ejemplo: no uses un elemen para tomar decisiones médicas críticas sin supervisión humana. No uses un elemen para generar contratos legales sin revisión de un abogado. El modelo puede producir texto que parece correcto pero contiene errores sutiles con consecuencias graves.
IAGO: [serio] Tercer límite: la IA no reemplaza el diseño del sistema. He visto decenas de proyectos que fallan porque el equipo asume que el modelo resolverá problemas de proceso. No lo hará. Si tu proceso es caótico, la IA amplificará el caos. Si tus datos son sucios, la IA producirá predicciones sucias. Si tus métricas de éxito no están claras, no sabrás si la IA está funcionando. El error común es pensar que la tecnología compensa la falta de claridad estratégica. No lo hace.

MARIA: [reflexivo] Entonces, ¿cuándo no debes usar IA?
<break time="500ms"/>
IAGO: [directo] No uses IA cuando no tienes datos suficientes o de calidad. No uses IA cuando no puedes tolerar errores y no tienes capacidad de supervisión humana. Es una regla de negocio bien diseñada, un dashboard bien construido, o un proceso manual optimizado.
