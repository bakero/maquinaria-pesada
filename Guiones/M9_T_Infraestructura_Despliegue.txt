# BLOQUE_1

MARÍA: [didáctico] Empecemos por la decisión que lo condiciona todo: dónde despliegas. Cloud público, on-premise, o híbrido. No es una decisión técnica. Es una decisión estratégica con implicaciones en coste, latencia, privacidad y capacidades que se extienden durante años.

IAGO: [explicativo] El cloud público, AWS, Azure, GCP, te da acceso inmediato a GPUs de última generación, escalado automático y servicios gestionados para todos los componentes de la infraestructura de IA. SageMaker, Azure ML, Vertex AI. Para la mayoría de organizaciones que empiezan con IA, es la opción correcta por defecto.

MARÍA: [curioso] ¿Y cuándo deja de ser la opción correcta?

IAGO: [serio] Cuando los datos no pueden salir de tu datacenter. Soberanía de datos. El GDPR en Europa, regulaciones sectoriales en banca y sanidad, contratos con clientes que prohíben explícitamente el procesamiento externo. En esos casos, on-premise no es una preferencia. Es un requisito legal.

MARÍA: [explicativo] Y hay un segundo escenario donde on-premise gana: cuando el volumen de inferencia es tan alto que el coste de cloud supera al de infraestructura propia. Eso tiene nombre técnico: el cloud crossover point. El punto exacto en que construir y operar tu propia GPU farm sale más barato que pagar por hora de cómputo externo.

IAGO: [reflexivo] ¿Y la estrategia más frecuente en la práctica?

MARÍA: [didáctico] Híbrida. APIs cloud para desarrollo y para los casos de uso de bajo volumen, donde la flexibilidad importa más que el coste marginal. Y modelos propios desplegados on-premise para los casos de uso de alto volumen o con datos sensibles. No es una elección binaria. Es una arquitectura de dos velocidades.


IAGO: [tenso] Y una vez decides dónde, viene la pregunta de cómo. Porque no hay una arquitectura de referencia para sistemas de IA. Hay cuatro patrones principales, cada uno con sus ventajas y sus límites.

MARÍA: [explicativo] El primero y más frecuente: API Gateway más LLM Backend. Una API REST que recibe la solicitud del usuario, la procesa, llama al modelo, y devuelve la respuesta. Simple, predecible, fácil de operar. Es el punto de partida correcto para el noventa por ciento de los casos.

IAGO: [conversacional] ¿Y cuándo ese patrón simple deja de ser suficiente?
# BLOQUE_2

MARÍA: [serio] Cuando necesitas escalar componentes de forma independiente. Ahí entra el patrón de microservicios. El servicio de retrieval para RAG, el servicio de inferencia del LLM, el orquestador de herramientas del agente: cada uno como servicio independiente con sus propios recursos, su propio escalado y su propio ciclo de vida.

IAGO: [didáctico] El tercer patrón es event-driven, orientado a eventos. Los eventos del negocio disparan workflows de IA de forma asíncrona. Un nuevo contrato subido activa automáticamente el análisis. Una alerta de sensor lanza el diagnóstico predictivo. Es mucho más robusto ante fallos y picos de tráfico que los patrones síncronos.

MARÍA: [reflexivo] Y el cuarto es serverless. Funciones Lambda de AWS o Azure Functions que se activan bajo demanda y escalan a cero cuando no hay tráfico. Ideal para cargas intermitentes, pero con un límite que hay que conocer: los tiempos máximos de ejecución pueden ser problemáticos para agentes de larga duración que necesitan iterar durante minutos.

IAGO: [firme] Ningún patrón es superior en abstracto. La elección depende del perfil de la carga, de los requisitos de latencia, del presupuesto de operaciones y del tamaño del equipo que va a mantener el sistema. Y en el módulo se trabaja cada uno con casos reales.


IAGO: [didáctico] Hablemos de seguridad. Porque en sistemas de IA en producción la seguridad tiene dimensiones que no aparecen en los proyectos de software tradicional y que generan vulnerabilidades muy específicas.

MARÍA: [explicativo] La primera: la gestión de secretos y credenciales. Los sistemas de IA necesitan acceder a múltiples servicios externos. APIs de LLM, bases de datos, APIs empresariales. Y cada uno de esos accesos necesita credenciales que deben protegerse con rigor. Los anti-patrones más peligrosos son los que más se ven: API keys hardcodeadas en el código, variables de entorno sin cifrar, archivos de configuración que acaban en el repositorio de git.

IAGO: [serio] La práctica correcta: usar un gestor de secretos. HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, Google Secret Manager. Credenciales cifradas, rotación automática, acceso auditado. Cada servicio tiene exactamente las credenciales que necesita para su función. Nada más.

MARÍA: [contundente] Ese principio tiene nombre: mínimo privilegio. Y en sistemas de IA tiene implicaciones directas. El agente de atención al cliente no debe tener acceso a las bases de datos financieras internas. El sistema de análisis de contratos no debe tener permisos de escritura en los sistemas de producción. Si un componente es comprometido o sufre un ataque de prompt injection exitoso, el daño potencial queda acotado.

IAGO: [conversacional] ¿Y cómo se protege el sistema contra el abuso externo?

# BLOQUE_3

MARÍA: [explicativo] Rate limiting como primera línea. Límite de solicitudes por usuario y por IP en una ventana de tiempo. Autenticación con JWT o API keys por usuario, que permite no solo control de acceso sino auditoría del uso y revocación granular si una credencial se compromete.

IAGO: [didáctico] Y el circuit breaker, que es crítico para la resiliencia. Cuando un componente externo, una API de LLM, una base de datos, falla o se degrada, el circuit breaker lo desconecta automáticamente del flujo principal. Evita que el fallo en cascada de ese componente tumbe el sistema completo. Es la diferencia entre un incidente localizado y una caída total.


MARÍA: [serio] Optimización de latencia. Porque en sistemas conversacionales, la percepción del usuario no depende de cuánto tarda el sistema en generar la respuesta completa. Depende de cuánto tarda en darle el primer token.

IAGO: [explicativo] Ese es el TTFT: Time To First Token, o tiempo hasta el primer token. Es la métrica crítica en experiencia de usuario para sistemas de IA conversacional. Y la técnica que más impacto tiene sobre ella es el streaming: devolver los tokens al usuario según se generan, sin esperar a que el modelo complete la respuesta. El sistema parece cuatro veces más rápido aunque tarde lo mismo en total.

MARÍA: [didáctico] Más allá del streaming, hay técnicas de aceleración de la inferencia que operan a nivel del modelo. La cuantización reduce la precisión de los parámetros de dieciséis bits a ocho bits o cuatro bits. Eso reduce el uso de memoria y aumenta la velocidad de inferencia con una pérdida de calidad mínima en la mayoría de casos de uso.

IAGO: [reflexivo] ¿Y qué hay de los frameworks de serving? Porque ahí hay una brecha enorme entre lo que da una implementación naive y lo que dan los frameworks especializados.

MARÍA: [explicativo] vLLM es el estado del arte actual para máximo throughput. Usa una técnica llamada PagedAttention que gestiona la memoria de la KV cache de forma similar a cómo los sistemas operativos gestionan la memoria virtual. El resultado es que puede servir muchos más requests simultáneos con la misma GPU. TGI, Text Generation Inference de Hugging Face, es la alternativa para escenarios donde la alta disponibilidad importa más que el throughput bruto.

IAGO: [conversacional] Y la selección de la GPU correcta, que es una decisión que la gente subestima.

MARÍA: [serio] Completamente. H100 y A100 para máxima velocidad cuando la latencia es el KPI crítico. L4 y T4 para máximo valor coste-rendimiento cuando el throughput importa más que la latencia individual. No todas las cargas necesitan la GPU más cara del mercado. Sobredimensionar el hardware es dinero quemado. Subdimensionarlo es latencia que destruye experiencia de usuario.

# BLOQUE_4


IAGO: [tenso] Llegamos a la técnica que más impacto tiene sobre el coste operativo de sistemas de IA en producción y que menos se enseña: el prompt caching.

MARÍA: [curioso] Explícamelo porque hay mucha confusión sobre qué es exactamente.

IAGO: [didáctico] El prompt caching funciona así: cuando el mismo prefijo de prompt se reutiliza en múltiples llamadas, el proveedor puede cachear los KV states del transformer para ese prefijo y evitar recalcularlos en cada llamada. El system prompt más el contexto estático de un asistente empresarial son siempre los mismos. Si el proveedor los ha cacheado, solo paga por procesar la parte que cambia: la pregunta del usuario.

MARÍA: [contundente] La reducción de coste es brutal. Entre un ochenta y un noventa por ciento en sistemas donde el system prompt es largo y constante. Un asistente con RAG donde el contexto de los documentos es estático es el caso ideal. Y lo tienen disponible Anthropic, OpenAI y Google en sus APIs principales. No es una característica de nicho. Es una palanca de coste que en producción puede cambiar completamente la ecuación económica de un proyecto.

IAGO: [serio] Y luego está el TCO, el coste total de propiedad. Porque cuando hablas con un directivo de finanzas, no le hablas de tokens. Le hablas de coste total. Y ese coste tiene componentes que muchos equipos técnicos olvidan presupuestar.

MARÍA: [explicativo] Coste de tokens: input tokens por precio por token más output tokens por precio por token. Ese coste puede variar cien veces entre un modelo frontier y un modelo pequeño para la misma tarea. Coste de infraestructura: servidores, instancias de GPU, bases de datos vectoriales, almacenamiento. Coste de observabilidad y monitorización. Y coste del equipo de ingeniería para el mantenimiento y la mejora continua, que en la práctica suele ser el componente más alto de todos.

IAGO: [didáctico] La gestión del TCO requiere tres capacidades que el módulo trabaja en detalle. Primero, monitorización del gasto de tokens en tiempo real con alertas cuando el gasto diario supera el presupuesto. Segundo, optimización del modelo por caso de uso: no todos los pasos del pipeline necesitan el modelo más caro. Muchos pasos de clasificación, extracción o resumen funcionan igual de bien con modelos diez veces más baratos.

MARÍA: [reflexivo] Y tercero, evaluación periódica de si los modelos más nuevos y baratos pueden reemplazar a los más caros sin pérdida de calidad medible. En 2025, los modelos de bajo coste han alcanzado la calidad que hace dos años solo tenían los frontier. El equipo que no revisa esa ecuación cada seis meses está pagando de más por defecto.


MARÍA: [natural] Piénsalo así. La infraestructura de IA es como el sistema de tuberías de un edificio. Puedes tener el mejor grifo del mercado, el más eficiente, el más preciso. Si las tuberías no tienen la presión correcta, si hay fugas en los codos, si el diámetro no aguanta el caudal en hora punta, el grifo da igual. Los usuarios no ven el modelo. Sienten la presión del agua. Y si baja, se van.
