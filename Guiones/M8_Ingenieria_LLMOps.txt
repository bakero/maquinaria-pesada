# BLOQUE_PANORAMA

IAGO: <didactico>Empecemos por el territorio. LLMOps es la extensión de MLOps para sistemas basados en modelos de lenguaje grandes. MLOps ya existía para machine learning clásico: modelos de regresión, árboles de decisión, redes neuronales tradicionales. Pero los LLMs traen problemas específicos que MLOps no cubría. El modelo ya no es un artefacto que entrenas y despliegas una vez. Es frecuentemente una API externa de OpenAI, Anthropic o Google que puede cambiar de comportamiento con cada actualización del proveedor. Los prompts son parte del código del sistema y deben versionarse con el mismo rigor que el código Python. La evaluación de calidad es mucho más compleja porque estás evaluando texto libre, no categorías predefinidas. Y los costes de inferencia son proporcionales al número de tokens que procesas y generas, lo cual introduce una dimensión económica que no existía en MLOps tradicional.</didactico>

MARIA: <esceptico>Entonces la diferencia fundamental es que en MLOps clásico tú controlas el modelo. Lo entrenas, lo despliegas, lo versiones. En LLMOps estás usando un servicio externo que puede cambiar sin avisarte.</esceptico>

IAGO: <firme>Exacto en parte. Pero incluso cuando haces fine-tuning de un modelo base y lo despliegas tú mismo, los problemas de LLMOps siguen ahí. Los prompts siguen siendo código vivo que cambia. La evaluación de texto libre sigue siendo más compleja que evaluar precisión en clasificación. Y el coste sigue siendo proporcional al volumen de tokens. Lo que LLMOps añade es la disciplina operacional para gestionar esa complejidad de forma sistemática.</firme>

IAGO: <explicativo>El módulo cubre tres bloques. Primero, el ciclo de vida de los modelos LLM en producción: versionado de modelos, prompts y datos, y la decisión estratégica de cuándo hacer fine-tuning versus quedarte con prompting. Segundo, observabilidad y monitorización: cómo ver qué está pasando en tu sistema LLM con logs, trazas y métricas, usando herramientas como LangSmith o Langfuse. Tercero, CI/CD y test-time compute: cómo automatizar la calidad con pipelines de integración continua específicos para IA, y cómo los modelos de razonamiento como o3 cambian el trade-off entre coste y calidad.</explicativo>

MARIA: <curioso>Y la pregunta clave que responde el módulo es: ¿por qué tantos proyectos de IA fallan en producción si el modelo funciona?</curioso>

IAGO: <contundente>Porque funcionar en un notebook no es lo mismo que funcionar en producción. En producción necesitas saber cuándo el modelo se degrada. Necesitas controlar los costes antes de que se disparen. Necesitas poder reproducir el estado del sistema de hace tres semanas cuando un usuario reporta un problema. Necesitas desplegar cambios sin romper lo que ya funciona. Y necesitas hacerlo de forma sostenible, no con heroicidades manuales cada vez que algo falla. LLMOps es la disciplina que hace eso posible.</contundente>

# BLOQUE_TEMAS_CLAVE

IAGO: <didactico>Primer concepto clave: versionado de modelos, prompts y datos. Sin versionado, es imposible saber qué produjo un cambio de comportamiento observado en producción. No puedes reproducir el estado del sistema en un momento anterior. No puedes deshacer un cambio que degrada el rendimiento. Para los LLMs, el versionado de prompts es especialmente crítico porque los prompts son parte del código del sistema. Un cambio en el prompt puede cambiar completamente el comportamiento del modelo. Ese cambio debe pasar por control de versiones en git, debe pasar por revisión de código igual que cualquier cambio de código Python, y debe testearse antes de desplegar en producción.</didactico>

MARIA: <analitica>Y cuando dices versionado de datos, ¿te refieres a los datos de entrenamiento si haces fine-tuning, o también a los datos de evaluación?</analitica>

IAGO: <explicativo>Ambos. Los datos de entrenamiento si haces fine-tuning, obviamente. Pero también los datos de evaluación: tu golden dataset de casos representativos que usas para medir si un cambio mejora o degrada el sistema. Ese dataset evoluciona con el tiempo: añades nuevos tipos de inputs que observas en producción, eliminas casos que ya no son representativos. Si no versionas ese dataset, no puedes comparar el rendimiento del sistema hoy con el rendimiento del sistema hace tres meses de forma justa, porque el dataset cambió. DVC, Data Version Control, es el estándar para esto: permite hacer commit de referencias a archivos grandes en el repositorio de git mientras almacena los archivos reales en almacenamiento externo como S3 o Google Cloud Storage.</explicativo>

MARIA: <directo>Segundo concepto: fine-tuning versus prompting. La decisión estratégica de cuándo entrenar el modelo y cuándo quedarte con prompts.</directo>

MARIA: <didactico>El prompting es más rápido: no requiere datos de entrenamiento ni tiempo de entrenamiento. Es más flexible: puedes cambiar el prompt en minutos y desplegar el cambio inmediatamente. Y es más económico en el corto plazo porque no pagas por entrenamiento. El fine-tuning produce mejor rendimiento para tareas bien definidas si tienes suficientes datos de entrenamiento: típicamente necesitas entre quinientos y mil ejemplos de alta calidad. Reduce el coste de inferencia porque el modelo fine-tuneado con instrucciones más cortas puede reemplazar prompts largos con muchos ejemplos. Y puede producir comportamientos que no son alcanzables con prompting: estilos muy específicos, formatos de output muy particulares, conocimiento de dominio especializado que el modelo base no tiene.</didactico>

IAGO: <reflexivo>La regla práctica que da el módulo es clara: empieza siempre con prompting. Si después de optimizar el prompt el rendimiento no es suficiente, entonces considera el fine-tuning. El fine-tuning es especialmente valioso en tres casos: cuando necesitas un comportamiento de formato muy específico que requiere muchos ejemplos en el prompt y el fine-tuning eliminaría esa necesidad, cuando quieres mejorar el rendimiento en tareas especializadas donde el modelo base tiene conocimiento limitado, y cuando tienes un volumen de inferencia muy alto y el modelo fine-tuneado con menor prompt puede ser más económico.</reflexivo>

IAGO: <didactico>Tercer concepto: observabilidad de sistemas LLM. La observabilidad tiene tres dimensiones. Logs: registro de todas las interacciones, prompts enviados, respuestas recibidas, metadatos de cada llamada. Trazas: seguimiento del flujo de una solicitud a través de todos los componentes del sistema. Si tienes un sistema RAG con recuperación de documentos, llamada al LLM, y ejecución de herramientas, la traza muestra todo ese flujo de principio a fin. Métricas: agregaciones de los logs y trazas en el tiempo. Latencia promedio, coste por solicitud, tasas de error, métricas de calidad del modelo. Sin estas tres dimensiones, operar un sistema LLM en producción es como conducir sin instrumentos: puedes ir hacia adelante pero no sabes si vas en la dirección correcta, a qué velocidad, o cuándo te vas a quedar sin combustible.</didactico>

MARIA: <esceptico>Y las métricas de calidad del modelo son las más difíciles de monitorizar porque tienes que evaluar la calidad del texto generado, no solo métricas técnicas como latencia o errores HTTP.</esceptico>

IAGO: <explicativo>Correcto. Las métricas proxy pueden ayudar: tasa de thumbs down de los usuarios, tasa de escalada a humanos, tasa de repetición de consultas que sugiere que la respuesta anterior no fue satisfactoria. Estas métricas pueden detectar degradación del modelo sin necesitar evaluación humana directa. Pero la técnica más potente es LLM-as-judge: usar un segundo LLM para evaluar la calidad de las respuestas del sistema sobre una muestra del cinco al diez por ciento del tráfico. Esto proporciona evaluaciones de calidad automáticas con mayor correlación con el juicio humano que las métricas proxy.</explicativo>

MARIA: <directo>Cuarto concepto: LangSmith y el ecosistema de observabilidad. LangSmith es la herramienta de observabilidad más integrada con el ecosistema LangChain y LangGraph. Se activa con dos variables de entorno y traza automáticamente todas las llamadas a LLMs, todas las recuperaciones RAG, y todas las ejecuciones de herramientas. Produce trazas visuales que muestran el flujo completo de cada solicitud. Langfuse es la alternativa open-source más popular, con soporte para múltiples frameworks y modelos, y la ventaja de poder desplegarse on-premise para mayor privacidad. Weights & Biases Traces integra la observabilidad de LLMs con el tracking de experimentos de entrenamiento, útil para equipos que hacen fine-tuning.</directo>

IAGO: <conversacional>Y la observabilidad no es solo para debugging cuando algo falla. Es para operación continua: ver cómo evoluciona el sistema, detectar degradación antes de que los usuarios se quejen, identificar oportunidades de optimización de costes.</conversacional>

IAGO: <didactico>Quinto concepto: CI/CD para sistemas de IA. El pipeline de integración continua para LLMs extiende el CI/CD estándar de software con checks específicos para IA. El pipeline de pull request en un sistema LLM incluye: tests unitarios del código Python del sistema con pytest, tests de integración de las herramientas del agente para verificar que las funciones de herramienta manejan correctamente los casos de borde, tests de seguridad del prompt con un golden dataset adversarial donde el sistema debe rechazar definitivamente esos prompts maliciosos, tests de calidad del LLM con un golden dataset de casos de uso representativos donde el sistema debe mantener o mejorar el rendimiento anterior, y análisis de costes para verificar que el cambio no aumenta el coste de tokens por encima de un umbral. Solo si todos estos checks pasan, el pull request puede mergearse.</didactico>

MARIA: <analitica>El test suite de evaluación del LLM es el componente más crítico y el más diferente al CI/CD estándar de software. Requiere un golden dataset de cincuenta a doscientos casos representativos con outputs de referencia, actualizados periódicamente con nuevos tipos de inputs observados en producción. Requiere evaluadores automáticos para medir la calidad, típicamente LLM-as-judge con rúbricas específicas para el caso de uso. Y requiere umbrales de regresión: si el rendimiento cae más de un porcentaje determinado respecto a la versión anterior, el pipeline falla. Este enfoque previene las regresiones accidentales de calidad que son una de las causas más comunes de degradación de sistemas LLM en producción.</analitica>

MARIA: <directo>Sexto concepto: test-time compute y modelos de razonamiento. Test-time compute es el paradigma de mejorar la calidad del modelo invirtiendo más computación durante la inferencia, en el tiempo de uso, no en el tiempo de entrenamiento. Los modelos de razonamiento como o3 de OpenAI y Claude con extended thinking usan este paradigma: en lugar de generar directamente la respuesta, generan un scratchpad de razonamiento que puede ser muy extenso antes de producir la respuesta final. Este razonamiento adicional en el tiempo de inferencia produce mejoras significativas en benchmarks de matemáticas, código, y razonamiento lógico.</directo>

IAGO: <explicativo>La implicación para LLMOps es que ahora hay un trade-off explícito entre coste-latencia y calidad que puedes ajustar en tiempo de ejecución. Para preguntas sencillas, usas el modelo estándar: rápido y barato. Para preguntas complejas, usas el modelo de razonamiento: más lento y caro pero más preciso. Esto introduce una nueva dimensión de optimización: clasificar las consultas por complejidad y enrutar cada consulta al modelo apropiado. Es lo que se llama modelos escalonados por complejidad.</explicativo>

# BLOQUE_LIMITES

MARIA: <tenso>Ahora hablemos de lo que falla. Primer límite: el versionado de prompts no resuelve el problema de que el modelo del proveedor puede cambiar sin avisarte. Tú versionas tu prompt. Pero si OpenAI actualiza GPT-4 y el nuevo modelo interpreta tu prompt de forma diferente, tu sistema se rompe aunque tu código no haya cambiado. La solución es versionar también el ID del modelo específico de la API, incluyendo versiones con fecha como gpt-4o-2024-08-06. Pero eso introduce un nuevo problema: esas versiones con fecha tienen fecha de caducidad. OpenAI eventualmente las depreca. Entonces tienes que migrar a la nueva versión, re-evaluar todo tu sistema, y potencialmente ajustar tus prompts. Es un coste operacional continuo que no existía en MLOps tradicional.</tenso>

IAGO: <reflexivo>Y no hay solución perfecta. Puedes usar modelos open-source que despliegas tú mismo para tener control total. Pero entonces asumes el coste de infraestructura y el coste de mantenimiento del modelo. O puedes usar APIs de terceros y aceptar que el modelo puede cambiar. La decisión depende de cuánto control necesitas versus cuánta complejidad operacional estás dispuesto a asumir.</reflexivo>

MARIA: <esceptico>Segundo límite: LLM-as-judge tiene sus propios sesgos. Estás usando un LLM para evaluar las respuestas de otro LLM. El juez puede tener preferencias sistemáticas: puede preferir respuestas más largas, o respuestas con cierto estilo, independientemente de si son mejores. El módulo no profundiza en esto, pero es un problema activo de investigación. La correlación con el juicio humano es mayor que las métricas proxy, sí. Pero no es perfecta. Y si optimizas tu sistema para maximizar la puntuación del juez, puedes estar optimizando para los sesgos del juez, no para la calidad real.</esceptico>

IAGO: <firme>La mitigación es calibrar el juez con evaluaciones humanas periódicas. Tomas una muestra de casos, los evalúas con el juez y con humanos, y mides la correlación. Si la correlación cae, ajustas las rúbricas del juez o cambias el modelo del juez. Pero sí, es un coste adicional de mantenimiento.</firme>

MARIA: <tenso>Tercer límite: el golden dataset se queda obsoleto. Los inputs que ves en producción evolucionan con el tiempo. Los usuarios encuentran nuevas formas de usar el sistema. Aparecen nuevos tipos de consultas que tu golden dataset no cubre. Si no actualizas el dataset periódicamente, tus tests de CI/CD siguen pasando pero el sistema se degrada en los casos nuevos que no estás testeando. El módulo dice que debes actualizar el dataset con nuevos tipos de inputs observados en producción. Pero no dice con qué frecuencia, ni cómo decides qué casos añadir, ni cómo evitas que el dataset crezca indefinidamente hasta que los tests tardan demasiado.</tenso>

IAGO: <conversacional>La práctica común es revisar el dataset cada trimestre, añadir los casos más representativos de los nuevos patrones observados, y eliminar casos redundantes o que ya no son relevantes. Pero sí, es trabajo manual. No hay automatización completa para esto.</conversacional>

MARIA: <ironico>Cuarto límite: test-time compute suena bien en teoría pero en la práctica tienes que clasificar las consultas por complejidad antes de enrutarlas al modelo apropiado. ¿Y cómo clasificas la complejidad de una consulta en lenguaje natural? Puedes usar otro LLM para clasificar. Pero entonces estás añadiendo latencia y coste de una llamada adicional antes de la llamada principal. O puedes usar heurísticas: longitud de la consulta, presencia de ciertas palabras clave. Pero las heurísticas son frágiles y se rompen con consultas adversariales.</ironico>

IAGO: <reflexivo>Y el trade-off no es binario. No es solo modelo rápido versus modelo de razonamiento. Puedes tener tres o cuatro modelos escalonados. Pero entonces la lógica de enrutamiento se vuelve más compleja. Y tienes que mantener métricas de calidad y coste para cada nivel. La complejidad operacional crece con el número de modelos que usas.</reflexivo>

# APLICACION_PRACTICA

IAGO: <conversacional>Ahora veamos cómo todo esto se aplica en un sistema real. Concretamente, en el sistema que está generando este podcast. Concretamente: ¿cómo gestionas la operación de un sistema LLM que tiene que producir cien episodios de podcast con calidad consistente, costes controlados, y capacidad de evolucionar sin romper lo que ya funciona?</conversacional>

MARIA: <didactico>El sistema de MaquinarIA Pesada es un caso de LLMOps en producción con restricciones reales. El sistema genera guiones de podcast usando Claude Sonnet, produce audio con ElevenLabs, y desde hace poco genera video con Kling. La arquitectura está documentada en BIBLIA_SISTEMA: un pipeline de tres etapas. Primera etapa: generación de guion con Claude a partir de un PDF resumen del módulo y de cuatro documentos vivos del proyecto que contienen decisiones técnicas y aprendizajes operativos. Segunda etapa: validación del guion contra una especificación normativa que incluye estructura obligatoria, longitudes mínimas por bloque, lista negra de interjecciones prohibidas, y reglas de reparto de palabras entre los dos presentadores. Tercera etapa: síntesis de audio con ElevenLabs y ensamblaje con música de fondo. El sistema versiona todo: los guiones generados se guardan en git, los audios finales se suben a un bucket de S3, y los PDFs fuente están versionados en el repositorio.</didactico>

IAGO: <esceptico>¿Y el versionado de prompts?</esceptico>

MARIA: <explicativo>El system prompt del generador de guiones está en el código del sistema, en el archivo de especificación PODCAST_M_SPEC. Tiene aproximadamente cinco kilobytes. Está dividido en reglas duras y reglas de tono. Las reglas duras incluyen estructura de bloques obligatorios, longitudes mínimas por intervención, lista negra de interjecciones, reparto de palabras por speaker. Las reglas de tono incluyen el perfil de cada presentador: Yago es explicador técnico, Maria es oyente exigente. Cualquier cambio en esa especificación pasa por git, pasa por revisión, y se testea antes de usarse para generar episodios. Eso es versionado de prompts aplicado.</explicativo>

MARIA: <directo>El problema operativo más grande que hemos tenido es la degradación silenciosa de calidad. En los primeros episodios, el sistema generaba interjecciones de validación tipo "exactamente", "claro que sí", "muy bien dicho". Esas interjecciones hacen que el diálogo suene artificial, tipo NotebookLM. Añadimos una lista negra de interjecciones prohibidas a la especificación. Pero el modelo seguía generándolas ocasionalmente. La solución fue un validador post-generación que parsea el guion, busca esas interjecciones, y si las encuentra, marca el guion como fallido. El episodio no se produce hasta que el guion pasa validación. Esto está documentado en PODCAST.md como una REGLA consolidada.</directo>

IAGO: <analitica>Ese validador es un ejemplo de CI/CD para LLMs. El pipeline de producción de un episodio incluye: generación del guion, validación estructural del guion contra la especificación, validación de la lista negra de interjecciones, síntesis de audio, validación de duración del audio contra el rango objetivo de catorce a dieciocho minutos. Si cualquier validación falla, el episodio no se publica. El sistema loguea el fallo, y el autor puede revisar qué falló y ajustar el prompt o regenerar. Eso previene regresiones de calidad.</analitica>

MARIA: <conversacional>¿Y la observabilidad? ¿Cómo ves qué está pasando en el sistema?</conversacional>

IAGO: <explicativo>El sistema loguea todas las llamadas a Claude: prompt enviado, respuesta recibida, tokens consumidos, coste de la llamada. Esos logs se guardan en archivos locales en la carpeta episodios. Para el pipeline de video, que usa Kling para generar clips, los logs incluyen también el ID de cada tarea de generación de video, el tiempo que tardó, y el coste. Esto permite hacer análisis post-mortem cuando algo falla: puedes ver exactamente qué prompt se envió, qué respuesta se recibió, y en qué punto del pipeline falló. No estamos usando LangSmith ni Langfuse porque el sistema no usa LangChain. Pero los logs cumplen la misma función: trazabilidad completa de cada episodio producido.</explicativo>

MARIA: <reflexivo>Y el coste. El módulo dice que los costes de inferencia son proporcionales al número de tokens procesados y generados. ¿Cómo gestionas eso?</reflexivo>

IAGO: <firme>El system prompt del generador tiene un presupuesto de tokens explícito: doscientos mil tokens de salida máximo. Eso limita el tamaño del guion generado. El modelo usado es Claude Sonnet, no Opus, porque Sonnet es más barato y suficientemente capaz para la tarea. Para el audio, ElevenLabs cobra por caracteres sintetizados. El sistema cuenta los caracteres del guion antes de sintetizar y estima el coste. Si el coste supera un umbral, el sistema alerta. Esto está documentado en PRIMERPODCAST.md: el coste de audio de un episodio típico es de uno a dos dólares. El coste de generación de guion con Claude es de aproximadamente cincuenta centavos por episodio. Son costes controlados porque el sistema los mide y los loguea en cada ejecución.</firme>

MARIA: <curioso>¿Y el fine-tuning? El módulo dice que empieces siempre con prompting. ¿Habéis considerado fine-tuning?</curioso>

IAGO: <conversacional>No. El sistema usa prompting exclusivamente. El prompt es largo y detallado, pero funciona. El rendimiento es suficiente: la mayoría de los guiones pasan validación a la primera. Cuando fallan, fallan por razones específicas: estructura incorrecta, longitud fuera de rango, interjecciones prohibidas. Esos fallos se resuelven ajustando el prompt, no haciendo fine-tuning. El fine-tuning sería útil si quisiéramos un estilo de escritura muy específico que requiere muchos ejemplos en el prompt. Pero no es el caso. El prompt actual tiene menos de diez ejemplos y el modelo generaliza bien.</conversacional>

MARIA: <directo>Y el aprendizaje operativo más importante: el sistema que produce este podcast no es estático. Evoluciona. Cada problema operativo que encontramos se documenta en PODCAST.md con marca de tiempo y se convierte en una regla consolidada en la especificación. Eso es LLMOps aplicado: el sistema aprende de sus propios fallos, y ese aprendizaje se codifica en reglas que previenen que el fallo se repita. No es automatización completa. Requiere intervención humana para documentar el problema y ajustar la especificación. Pero es sostenible. El sistema mejora con el tiempo sin heroicidades manuales en cada episodio.</directo>

IAGO: <reflexivo>Y eso conecta con el concepto central del módulo: LLMOps cierra la brecha entre experimentos y sistemas en producción. El sistema de MaquinarIA Pesada funciona en producción porque aplica las prácticas de LLMOps: versionado de prompts y código, validación automática de calidad, observabilidad con logs completos, control de costes con presupuestos explícitos, y evolución continua documentada. Sin esas prácticas, el sistema sería frágil. Con esas prácticas, es sostenible.</reflexivo>
