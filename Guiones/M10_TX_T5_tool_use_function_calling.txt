# BLOQUE_PANORAMA

IAGO: [didactico] Hablemos claro de qué es el function calling. Es el mecanismo que permite a un modelo de lenguaje grande expresar que necesita ejecutar una función específica para completar una tarea. Imagina que tienes un asistente muy inteligente pero que vive encerrado en una habitación sin ventanas. Puede razonar sobre cualquier cosa que le preguntes, pero no puede salir a buscar información actualizada ni tocar nada del mundo exterior. El function calling es la puerta que le permite salir y usar herramientas reales: buscar en internet, consultar una base de datos, enviar un correo, ejecutar código Python. Sin esa puerta, el asistente solo puede trabajar con lo que ya tiene en su memoria de entrenamiento.

MARIA: [curioso] ¿Y cómo sabe el modelo cuándo necesita usar una herramienta?

IAGO: [explicativo] El desarrollador define un conjunto de herramientas disponibles con sus descripciones y parámetros. Cada herramienta tiene un nombre descriptivo, una explicación de qué hace y un esquema JSON que especifica qué argumentos necesita. Cuando el modelo recibe una pregunta del usuario, analiza si puede responderla con su conocimiento interno o si necesita una herramienta. Si decide que necesita una herramienta, no genera texto libre diciendo "necesito buscar en internet". Genera un bloque estructurado en JSON con el nombre exacto de la función y los argumentos válidos según el esquema que definiste. Eso elimina el parsing ambiguo y los errores de interpretación.

IAGO: [firme] Por qué importa esto en sistemas empresariales. Un agente sin herramientas es solo un chatbot avanzado. Un agente con herramientas bien diseñadas puede automatizar flujos de trabajo completos: analizar datos financieros consultando APIs externas, gestionar tickets en sistemas como Zendesk o Jira, ejecutar código para procesar datasets, enviar notificaciones por correo o Slack. La calidad del conjunto de herramientas que diseñes determina la capacidad real del sistema agéntico. No es el modelo lo que limita, es el set de herramientas.
MARIA: [reflexivo] Hay algo que me genera curiosidad en este punto. ¿Cómo conecta esto con lo que acabamos de ver? Porque la secuencia conceptual importa mucho aquí. No quiero perder el hilo de la progresión lógica.

# BLOQUE_COMO

IAGO: [didactico] Vamos al mecanismo técnico. El ciclo de function calling tiene cuatro pasos bien definidos. Primero, el desarrollador define las herramientas con sus esquemas JSON: nombre de la función, descripción de qué hace, parámetros con sus tipos y si son obligatorios u opcionales. Segundo, el modelo recibe el mensaje del usuario más la lista completa de herramientas disponibles. Analiza la consulta y decide si necesita usar alguna herramienta o puede responder directamente. Tercero, si decide usar una herramienta, devuelve un mensaje especial con el nombre de la función y los argumentos en formato JSON estructurado. Cuarto, el desarrollador ejecuta esa función con los argumentos proporcionados y devuelve el resultado al modelo como tool result. El modelo procesa el resultado y decide si continuar con otra herramienta o producir la respuesta final al usuario.

MARIA: [reflexivo] ¿Y qué pasa cuando el modelo necesita usar varias herramientas a la vez?

MARIA: [explicativo] Ahí entra el parallel tool calling. Los modelos modernos como GPT cuatro o Claude tres punto cinco pueden llamar a múltiples herramientas simultáneamente en un solo turno. Imagina que le preguntas a un agente financiero cómo está valorada una empresa respecto a sus competidores. El modelo puede decidir llamar en paralelo a tres herramientas: una que obtiene los datos financieros de la empresa, otra que busca el benchmark del sector, y una tercera que busca noticias recientes sobre esa empresa. Las tres llamadas se ejecutan al mismo tiempo con asyncio o threading. El resultado es que la latencia total es igual a la de la llamada más lenta, no la suma de las tres. Eso reduce significativamente el tiempo de respuesta en agentes que necesitan múltiples búsquedas independientes.
IAGO: [explicativo] Hay un matiz importante ahí. Los datos de producción raramente cumplen las condiciones del laboratorio. El modelo tiene que ser robusto a esa variabilidad. Y eso determina qué arquitectura es viable en la práctica.

MARIA: [serio] Ahora hablemos de los riesgos técnicos. El tool hallucination es uno de los problemas más comunes. Ocurre cuando el modelo invoca una herramienta con argumentos incorrectos: por ejemplo, pasa una cadena de texto donde se espera un número. O peor, invoca una herramienta que no existe en el conjunto disponible porque confunde nombres similares. O genera resultados inventados para una herramienta sin haberla llamado realmente. Las mitigaciones son esquemas JSON estrictos que verifican que los argumentos son del tipo correcto, validación de argumentos en el cliente antes de ejecutar la función, y logging completo de todas las llamadas con sus argumentos para auditoría posterior.
IAGO: [reflexivo] La pregunta que planteas toca algo crítico del diseño. El contexto cambia todo en estos sistemas distribuidos. Lo que funciona con datos limpios no funciona igual con inputs variables. El umbral de confianza es el parámetro que más afecta al resultado final.

MARIA: [analitica] ¿Y el tema de ejecutar código arbitrario?

IAGO: [tenso] Permitir que un agente ejecute código arbitrario es una de las capacidades más poderosas pero también más peligrosas. Un sandbox de código aísla la ejecución para evitar que el código generado por el modelo pueda dañar el sistema host. Sin sandbox, el código podría acceder a archivos del sistema, hacer llamadas de red no autorizadas, consumir todos los recursos disponibles o ejecutar código malicioso. Las opciones de sandbox son contenedores Docker efímeros, microVMs como Firecracker, servicios de ejecución de código en la nube como E2B o Modal, o el intérprete de código integrado de la API de OpenAI. El sandbox es obligatorio, no opcional.

La base de conocimiento incluye más de cien mil documentos internos: informes de análisis, presentaciones de producto, políticas de compliance. Los asesores hacen preguntas en lenguaje natural y el sistema recupera los documentos relevantes y genera un resumen contextualizado. El resultado documentado es que el tiempo de búsqueda de información pasó de horas a minutos. El socio tecnológico fue OpenAI con GPT cuatro. Esto no es un piloto, es producción a escala en uno de los bancos de inversión más grandes del mundo.

MARIA: [analitica] Otro caso verificado es Harvey AI en el sector legal. Es una startup fundada en dos mil veintidós especializada en IA para bufetes. La tecnología base es un modelo de lenguaje con fine tuning sobre corpus jurídico e integración con bases de datos legales como Westlaw y LexisNexis. Durante dos mil veintitrés, más del noventa y siete por ciento de los grandes bufetes de Estados Unidos estuvieron en lista de espera. Los casos de uso son redacción de contratos, due diligence, investigación jurídica y análisis de precedentes. El resultado reportado por usuarios piloto es una reducción de entre treinta y setenta por ciento del tiempo en tareas de revisión documental.

IAGO: [reflexivo] ¿Qué herramientas específicas usan esos agentes?

MARIA: [explicativo] En el caso de Harvey, las herramientas incluyen búsqueda semántica en bases de datos jurídicas, extracción de cláusulas específicas de contratos, comparación de precedentes judiciales y generación de borradores de documentos legales. Cada herramienta tiene un esquema JSON preciso que especifica qué tipo de consulta acepta y qué formato de datos devuelve. La clave está en que las herramientas no son genéricas: están diseñadas específicamente para el flujo de trabajo de un abogado corporativo.

MARIA: [serio] Ahora el reto documentado más común. El tool hallucination que mencionaba Yago es un problema real en producción. Cuando defines demasiadas herramientas, el modelo tiene dificultades para seleccionar la correcta. La regla práctica que reportan equipos de ingeniería es que entre cinco y quince herramientas bien definidas son más efectivas que cincuenta herramientas ambiguamente definidas. El modelo puede confundir herramientas similares o elegir la menos adecuada si las descripciones no son suficientemente precisas sobre cuándo usar cada una.
IAGO: [directo] Bien apuntado. Déjame añadir la perspectiva técnica aquí. Hay una capa de implementación que cambia cómo se interpreta lo que acabas de describir. El mecanismo interno es lo que determina la robustez real del sistema.

MARIA: [contundente] Y está el riesgo de seguridad con ejecución de código. Dar a un agente una herramienta que ejecute código sin sandbox es un riesgo crítico. El código generado por el modelo puede hacer rm menos rf barra, exfiltrar datos o consumir todos los recursos del sistema. Los equipos de plataforma que implementan agentes con capacidad de ejecución de código siempre usan contenedores efímeros o servicios especializados. No es paranoia, es ingeniería responsable.
