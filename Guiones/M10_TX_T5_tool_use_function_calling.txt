# BLOQUE_PANORAMA

IAGO: [didactico] Function calling es el mecanismo que permite a un modelo de lenguaje invocar funciones externas de forma estructurada. Sin herramientas, un modelo solo puede razonar sobre lo que ya sabe. Con function calling, puede buscar en la web, consultar bases de datos, ejecutar código, enviar emails o hacer llamadas a APIs empresariales. Es la diferencia entre un sistema que te explica cómo buscar información financiera y un sistema que la busca por ti y te la entrega procesada.

MARIA: [curioso] ¿Y cómo sabe el modelo qué herramienta usar?

IAGO: [explicativo] El desarrollador define un catálogo de herramientas disponibles. Cada herramienta tiene un nombre descriptivo, una descripción de qué hace y cuándo usarla, y un esquema JSON que especifica los parámetros que necesita. El modelo recibe ese catálogo junto con la pregunta del usuario, decide si necesita usar una herramienta, y devuelve un bloque estructurado con el nombre de la función y los argumentos. El desarrollador ejecuta esa función en el mundo real y devuelve el resultado al modelo. El modelo procesa el resultado y decide si necesita otra herramienta o si ya puede responder.

MARIA: [esceptico] Entonces el modelo no ejecuta nada directamente.

IAGO: [firme] El modelo solo expresa intención. Dice "necesito ejecutar la función buscar noticias de empresa con el argumento Tesla". El código del desarrollador es quien realmente hace la llamada a la API de noticias. Esto es crítico por seguridad: el modelo nunca tiene acceso directo a tus sistemas. Siempre hay un intermediario que valida y ejecuta.

# BLOQUE_COMO

IAGO: [didactico] El ciclo de function calling tiene cuatro pasos. Primero, defines las herramientas con sus esquemas. Segundo, el modelo recibe el mensaje del usuario más la lista de herramientas disponibles. Tercero, si el modelo decide usar una herramienta, devuelve un mensaje especial con el nombre de la función y los argumentos en formato JSON. Cuarto, ejecutas la función con esos argumentos y devuelves el resultado como entrada nueva al modelo. El modelo procesa el resultado y decide si continúa con otra herramienta o produce la respuesta final.

MARIA: [curioso] ¿Qué pasa si el modelo pide una herramienta que no existe?

IAGO: [serio] Eso se llama tool hallucination. Ocurre cuando el modelo invoca una herramienta con argumentos incorrectos, invoca una herramienta que no está en el catálogo, o peor, genera resultados inventados sin haber llamado a la herramienta. Se mitiga con esquemas JSON estrictos que validan los tipos de datos, con validación de argumentos en el cliente antes de ejecutar, y con logging completo de todas las llamadas para auditoría.

MARIA: [analitica] Hablemos del diseño de herramientas. ¿Qué hace que una herramienta sea buena?

IAGO: [explicativo] Tres cosas. Primera, nombres descriptivos que el modelo pueda entender. Buscar web es mejor que herramienta uno. Segunda, descripciones claras de qué hace la herramienta, cuándo usarla y qué devuelve. Tercera, esquemas JSON precisos con tipos, restricciones y ejemplos cuando sean útiles. Una herramienta bien diseñada tiene comportamiento predecible: los mismos argumentos producen el mismo resultado.

MARIA: [directo] ¿Y cuántas herramientas puede manejar un agente?

IAGO: [contundente] Entre cinco y quince herramientas bien definidas son más efectivas que cincuenta herramientas ambiguas. Un agente con cincuenta herramientas tiene dificultades para seleccionar la correcta. El modelo puede confundir herramientas similares o elegir la menos adecuada. La calidad del catálogo importa más que el tamaño.

MARIA: [curioso] Los modelos modernos pueden llamar a varias herramientas a la vez, ¿no?

IAGO: [didactico] Sí. Eso se llama parallel tool calling. En lugar de llamar primero a buscar web, esperar el resultado y después llamar a obtener datos financieros, el modelo puede decidir llamar a ambas simultáneamente. Reduces la latencia de forma significativa. La implementación requiere que el desarrollador ejecute las llamadas paralelas con asyncio o threading y devuelva todos los resultados juntos. La latencia total es igual a la de la llamada más lenta, no a la suma de las dos.

MARIA: [reflexivo] Y cuando hablamos de ejecutar código, ¿cómo funciona eso?

IAGO: [serio] Permitir que un agente ejecute código arbitrario es una de las capacidades más poderosas pero también más peligrosas. Un sandbox de código aísla la ejecución para evitar que el código generado por el modelo pueda dañar el sistema host. Las opciones son contenedores Docker efímeros, microVMs como Firecracker, o servicios de ejecución de código en la nube como E2B o Modal. El código se ejecuta en un entorno aislado sin acceso a archivos del sistema, sin llamadas de red no autorizadas, y con límites de recursos.

# BLOQUE_REALIDAD

MARIA: [analitica] Veamos cómo esto funciona en empresas reales. Morgan Stanley tiene un sistema de RAG para sus asesores financieros que usa function calling. La base de conocimiento tiene más de cien mil documentos internos: informes de análisis, presentaciones de producto, políticas de compliance. Los asesores hacen preguntas en lenguaje natural y el sistema recupera los documentos relevantes y genera un resumen contextualizado. El resultado documentado: reducción del tiempo de búsqueda de información de horas a minutos. El socio tecnológico es OpenAI con GPT-4.

IAGO: [directo] El modelo no busca directamente en los documentos.

MARIA: [firme] No. El modelo llama a una herramienta de búsqueda vectorial que devuelve los fragmentos relevantes. El modelo procesa esos fragmentos y construye la respuesta. La herramienta hace el trabajo pesado de recuperación. El modelo hace el trabajo de síntesis.

MARIA: [reflexivo] Otro caso: agentes de análisis financiero con parallel tool calling. El modelo recibe la pregunta "¿Cómo está valorada la empresa X respecto a sus competidores en el sector Y?" y decide llamar en paralelo a tres herramientas: obtener datos financieros de empresa X, obtener benchmark del sector Y, y buscar noticias recientes de empresa X. Los tres resultados se devuelven simultáneamente y el modelo los usa para construir el análisis comparativo. La latencia total es igual a la de la llamada más lenta, no a la suma de las tres.

IAGO: [pausado] Sin parallel calling, ese análisis llevaría el triple de tiempo.

MARIA: [contundente] Y en contextos empresariales donde cada segundo cuenta, esa diferencia es crítica. Ahora hablemos de riesgos. IBM Watson Health es el caso canónico de fracaso en IA para oncología. La promesa inicial era diagnosticar y recomendar tratamientos de cáncer mejor que oncólogos. MD Anderson Cancer Center canceló el proyecto en dos mil diecisiete tras invertir sesenta y dos millones de dólares. Memorial Sloan Kettering también redujo uso. La causa raíz documentada: el sistema fue entrenado con datos sintéticos y casos hipotéticos, no con pacientes reales. Las recomendaciones eran inconsistentes con protocolos médicos reales.

MARIA: [serio] La lección es clara: las expectativas sobre IA deben calibrarse con la calidad de los datos de entrenamiento. El hype no garantiza el rendimiento. Y cuando hablamos de function calling, esto se traduce en que las herramientas deben estar diseñadas para el mundo real, no para escenarios ideales. Una herramienta que funciona en el laboratorio puede fallar en producción si no contempla los casos edge, los errores de red, o las inconsistencias de datos.
MARIA: [reflexivo] Hay algo que me genera curiosidad en este punto. ¿Cómo conecta esto con lo que acabamos de ver? Porque la secuencia conceptual importa mucho aquí. No quiero perder el hilo de la progresión lógica.

MARIA: [analitica] Un último caso: Lemonade, la aseguradora digital, usa IA para procesamiento automatizado de siniestros. El tiempo de resolución de siniestros es de tres segundos en casos simples. El récord documentado: un siniestro aprobado y pagado en dos segundos. La IA analiza el vídeo de declaración del cliente, cruza datos con bases de datos de fraude, verifica la póliza y aprueba el pago. Cada uno de esos pasos es una llamada a una herramienta diferente. La orquestación es lo que hace que el sistema funcione.
