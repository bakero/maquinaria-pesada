# BLOQUE_PANORAMA

IAGO: [didactico] Tool use es el mecanismo que permite a un agente de inteligencia artificial interactuar con el mundo externo. Sin herramientas, un modelo de lenguaje solo puede razonar sobre lo que ya sabe. Con herramientas, puede buscar información actualizada en internet, ejecutar código Python, consultar bases de datos, enviar emails, o hacer llamadas a APIs externas. Es la diferencia entre un asistente que piensa y uno que hace. Function calling es el nombre técnico del mecanismo que hace esto posible. En lugar de que el modelo genere texto libre diciendo "ahora voy a buscar en Google", el modelo devuelve un bloque estructurado en JSON con el nombre exacto de la función que necesita ejecutar y los argumentos válidos. El desarrollador recibe ese JSON, ejecuta la función real, y devuelve el resultado al modelo para que continúe su razonamiento. Es un ciclo de cuatro pasos que se repite hasta que el agente completa la tarea.

MARIA: [curioso] ¿Y qué pasa si el modelo inventa una función que no existe?

IAGO: [serio] Ese es uno de los errores más comunes. Se llama tool hallucination. El modelo puede inventar nombres de herramientas, pasar argumentos del tipo incorrecto, o incluso generar resultados falsos sin haber llamado a la herramienta. Por eso el diseño del set de herramientas es crítico. Cada herramienta necesita un nombre descriptivo, una descripción clara de qué hace y cuándo usarla, y un schema JSON estricto que defina los tipos de los parámetros. Si defines una herramienta llamada "fetch data" con la descripción "obtiene datos", el modelo no tiene forma de saber cuándo usarla. Pero si la llamas "search company news" y describes que busca noticias recientes de una empresa específica usando su nombre o ticker, y que no debe usarse para información histórica anterior a dos mil veinte, el modelo tiene contexto suficiente para decidir correctamente.

MARIA: [analitica] ¿Cuántas herramientas puede manejar un agente de forma efectiva?

IAGO: [firme] La regla práctica es entre cinco y quince herramientas bien definidas. Un agente con cincuenta herramientas disponibles tiene dificultades para seleccionar la correcta. El modelo puede confundir herramientas similares o elegir la menos adecuada. Es mejor tener diez herramientas con descripciones precisas que cincuenta herramientas ambiguas.

# BLOQUE_COMO

IAGO: [explicativo] El ciclo de function calling funciona en cuatro pasos. Primero, el desarrollador define las herramientas con sus schemas JSON: nombre, descripción, y parámetros con sus tipos. Segundo, el modelo recibe el mensaje del usuario más la lista de herramientas disponibles y decide si necesita usar alguna. Tercero, si el modelo decide usar una herramienta, devuelve un mensaje especial. En Anthropic se llama tool use block, en OpenAI se llama function call. Ese mensaje contiene el nombre de la función y los argumentos en formato JSON. Cuarto, el desarrollador ejecuta la función con esos argumentos y devuelve el resultado como tool result al modelo. El modelo procesa el resultado y decide si necesita llamar a otra herramienta o si ya puede producir la respuesta final.

MARIA: [curioso] ¿Puede el modelo llamar a varias herramientas a la vez?

MARIA: [didactico] Sí, y eso se llama parallel tool calling. Los modelos modernos como GPT cuatro y Claude tres punto cinco soportan llamadas paralelas a múltiples herramientas en un solo turno. Imagina que el agente necesita buscar noticias de una empresa y al mismo tiempo consultar sus datos financieros. En lugar de llamar primero a search web, esperar el resultado, y después llamar a get financial data, el modelo puede decidir llamar a ambas simultáneamente. Esto reduce significativamente la latencia. La implementación requiere que el desarrollador ejecute las llamadas paralelas con asyncio o threading y devuelva todos los resultados juntos. La latencia total es igual a la de la llamada más lenta, no a la suma de todas. En agentes que hacen múltiples búsquedas independientes, esto puede reducir el tiempo de respuesta a la mitad.

IAGO: [serio] ¿Y cómo se previene que el modelo ejecute código peligroso?

IAGO: [firme] Ahí entra el sandbox de ejecución de código. Permitir que un agente ejecute código arbitrario es una de las capacidades más poderosas pero también más peligrosas. Un sandbox aísla la ejecución para evitar que el código generado por el modelo pueda dañar el sistema host. Sin sandbox, el código podría acceder a archivos del sistema, hacer llamadas de red no autorizadas, consumir recursos excesivos, o ejecutar comandos maliciosos. Las opciones de sandbox son contenedores Docker efímeros, micro máquinas virtuales como Firecracker, o servicios de ejecución de código en la nube como E dos B o Modal. OpenAI tiene un intérprete de código integrado en su API que ya viene con sandbox incluido. La regla dura es: si tu agente puede ejecutar código, ese código debe correr en un entorno aislado sin acceso al sistema principal.
MARIA: [reflexivo] Hay algo que me genera curiosidad en este punto. ¿Cómo conecta esto con lo que acabamos de ver? Porque la secuencia conceptual importa mucho aquí. No quiero perder el hilo de la progresión lógica.

IAGO: [reflexivo] ¿Qué diferencia hay entre herramientas con estado y sin estado?

MARIA: [analitica] Las herramientas sin estado producen siempre el mismo resultado con los mismos argumentos. Una búsqueda web o una consulta a base de datos son sin estado. Son las más simples de usar y de testear. Las herramientas con estado modifican el estado del sistema externo y pueden tener efectos secundarios: enviar un email, crear un registro en la base de datos, publicar un mensaje en Slack. Las herramientas con estado requieren mayor cuidado. El agente no debe llamarlas repetidamente si ya las ejecutó, y deben tener mecanismos de rollback para los casos donde la acción fue incorrecta. Por ejemplo, si el agente envía un email al cliente equivocado, necesitas poder revertir esa acción o al menos registrarla para auditoría.

# BLOQUE_REALIDAD

MARIA: [conversacional] Veamos cómo esto funciona en empresas reales. Morgan Stanley implementó un sistema de RAG con function calling para sus asesores financieros. El sistema tiene acceso a una base de conocimiento con más de cien mil documentos internos: informes de análisis, presentaciones de producto, políticas de compliance. Los asesores hacen preguntas en lenguaje natural y el sistema recupera los documentos relevantes usando herramientas de búsqueda y genera un resumen contextualizado. El resultado documentado es que el tiempo de búsqueda de información pasó de horas a minutos. El socio tecnológico fue OpenAI con GPT cuatro. Esto se presentó públicamente en el OpenAI DevDay de dos mil veintitrés.

IAGO: [curioso] ¿Qué herramientas específicas usa ese sistema?

MARIA: [claro] Herramientas de búsqueda semántica en la base documental, herramientas de filtrado por tipo de documento o fecha, y herramientas de extracción de citas específicas. Cada herramienta tiene un schema JSON estricto. Si el asesor pregunta sobre un producto financiero específico, el modelo decide qué herramientas usar en paralelo para obtener la información más relevante en una sola llamada.

MARIA: [analitica] Otro caso es el de agentes de análisis financiero con parallel tool calling. El modelo recibe la pregunta "¿Cómo está valorada la empresa equis respecto a sus competidores en el sector y griega?". El modelo decide llamar en paralelo a tres herramientas: get company financials para la empresa equis, get sector benchmark para el sector y griega, y search news para noticias recientes de la empresa equis. Los tres resultados se devuelven simultáneamente y el modelo los usa para construir el análisis comparativo. La latencia total es la de la llamada más lenta, no la suma de las tres.
IAGO: [directo] Bien apuntado. Déjame añadir la perspectiva técnica aquí. Hay una capa de implementación que cambia cómo se interpreta lo que acabas de describir. El mecanismo interno es lo que determina la robustez real del sistema.

MARIA: [serio] Pero no todo es éxito. Uno de los riesgos más documentados es el tool hallucination. Según estudios recientes de Gartner, entre el veinte y el treinta por ciento de las llamadas a herramientas en sistemas agénticos mal configurados contienen errores de validación: argumentos del tipo incorrecto, herramientas inexistentes invocadas, o resultados inventados. Las mitigaciones son schemas JSON estrictos que el structured output del modelo verifica automáticamente, validación de argumentos en el cliente antes de ejecutar la herramienta, y logging de todas las llamadas a herramientas con sus argumentos para auditoría posterior.

IAGO: [reflexivo] ¿Qué pasa con la ejecución de código en producción?

MARIA: [firme] Ahí el sandbox es obligatorio. Un caso real es el de agentes de análisis de datos que pueden escribir y ejecutar código Python en un sandbox como E dos B. El agente puede procesar datasets con pandas y numpy, generar visualizaciones con matplotlib, calcular estadísticas complejas, y ejecutar modelos de machine learning. El código se ejecuta en un contenedor aislado sin acceso al sistema del usuario. Sin sandbox, un error del modelo o un prompt injection podría ejecutar código malicioso en el servidor de producción. La regla de seguridad es: si el agente puede ejecutar código arbitrario, ese código debe correr en un entorno aislado siempre.

MARIA: [conversacional] Un último caso: agentes de ITSM que integran Zendesk y Jira. El agente tiene herramientas para buscar tickets similares en Zendesk, consultar el historial de incidentes en Jira, crear un ticket nuevo, asignarlo al equipo correcto, y notificar al usuario por email. Cada herramienta tiene un schema JSON estricto. Si hay un error de validación, el sistema devuelve al agente un mensaje claro sobre qué corregir. Esto reduce el tool hallucination porque el modelo recibe feedback inmediato cuando se equivoca.
