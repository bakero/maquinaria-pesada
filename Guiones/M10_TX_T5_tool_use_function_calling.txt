Sin function calling, un LLM solo puede razonar sobre el mundo. Con function calling, puede actuar en él: consultar bases de datos, ejecutar código Python, enviar emails, llamar a APIs de terceros. Es la diferencia entre un asistente que te dice cómo hacer algo y uno que lo hace por ti. Hoy explicamos el mecanismo técnico que permite a los agentes de IA interactuar con sistemas reales. # BLOQUE_QUE

IAGO: [didactico] Empecemos por la definición operativa. Function calling es el mecanismo técnico que permite a un modelo de lenguaje expresar que necesita ejecutar una función específica. No genera texto libre diciendo "llama a la función buscar_web con el argumento empresa X". Genera un bloque JSON estructurado con el nombre exacto de la función y los argumentos válidos según el schema que tú, como desarrollador, has definido. El desarrollador ejecuta esa función en su sistema y devuelve el resultado al modelo para que continúe su razonamiento. Es un ciclo: el modelo pide, tú ejecutas, el modelo procesa el resultado.

MARIA: ¿Y por qué no puede el modelo ejecutar directamente la función?

IAGO: [explicativo] Porque el modelo no tiene acceso a tu base de datos, a tu API de correo, a tu sistema de archivos. El modelo vive dentro de la infraestructura del proveedor: OpenAI, Anthropic, Google. Tú vives en tu empresa, con tus sistemas. Function calling es el puente: el modelo te dice qué hacer en formato estructurado, tú lo haces en tu entorno, y le devuelves el resultado. Sin ese puente, el modelo solo puede hablar sobre el mundo. Con ese puente, puede actuar en él.

MARIA: ¿Qué diferencia hay entre tool use y function calling?

IAGO: [claro] Tool use es el concepto: las herramientas que tiene disponibles un agente. Function calling es la implementación técnica: cómo el modelo invoca esas herramientas a través de la API. Cuando dices "mi agente tiene acceso a búsqueda web, ejecución de código y consulta de base de datos", estás hablando de tool use. Cuando dices "el modelo devolvió un bloque JSON con name: search_web y arguments: query: 'resultados financieros Tesla 2024'", estás hablando de function calling.

# BLOQUE_COMO

IAGO: [didactico] El ciclo de function calling tiene cuatro pasos. Primero: defines las herramientas con schemas JSON. Cada herramienta tiene un nombre, una descripción de qué hace y cuándo usarla, y un input_schema que especifica los parámetros: tipos, si son obligatorios, restricciones. Segundo: envías el mensaje del usuario más la lista de herramientas disponibles al modelo. Tercero: el modelo decide si necesita usar una herramienta. Si decide usarla, devuelve un mensaje especial, un bloque tool_use en Anthropic o function_call en OpenAI, con el nombre de la función y los argumentos. Cuarto: tú ejecutas la función con esos argumentos y devuelves el resultado como tool_result al modelo. El modelo procesa el resultado y decide si necesita otra herramienta o si ya puede producir la respuesta final.

MARIA: [conversacional] Vale, pero ¿qué pasa si el modelo necesita llamar a tres herramientas distintas para responder? ¿Tiene que ir una por una?

IAGO: No. Los modelos modernos, GPT-4, Claude 3.5, soportan parallel tool calling. El modelo puede decidir llamar a múltiples herramientas en un solo turno. Por ejemplo, si le preguntas "¿cómo está valorada la empresa X respecto a sus competidores en el sector Y?", puede llamar simultáneamente a get_company_financials para la empresa X, get_sector_benchmark para el sector Y, y search_news para noticias recientes de X. Tú ejecutas las tres llamadas en paralelo con asyncio o threading y devuelves todos los resultados juntos. La latencia total es igual a la de la llamada más lenta, no a la suma de las tres.

MARIA: [analitica] Y el diseño de las herramientas, ¿qué hace que una herramienta esté bien diseñada?

IAGO: [explicativo] Cuatro cosas. Primero, nombres descriptivos que el modelo pueda entender: search_web, get_company_financials, send_email son mejores que tool1, fetch_data, action3. Segundo, descripciones claras de qué hace la herramienta, cuándo usarla y qué devuelve. No basta con decir "obtiene datos". Tienes que decir "busca noticias recientes sobre una empresa específica usando su nombre o ticker, útil para información actualizada sobre eventos recientes como adquisiciones o resultados financieros, no usar para información histórica anterior a 2020". Tercero, schemas JSON precisos con tipos, restricciones y ejemplos cuando son útiles. Cuarto, comportamiento predecible: la misma herramienta con los mismos argumentos debe producir el mismo resultado, o documentar explícitamente si es no determinista.

MARIA: [conversacional] Hablemos de ejecución de código. Es una de las capacidades más potentes de un agente, pero también de las más peligrosas.

IAGO: [serio] Permitir que un agente ejecute código arbitrario es exactamente eso: la capacidad más poderosa y la más peligrosa. Un sandbox de código aísla la ejecución para evitar que el código generado por el LLM pueda dañar el sistema host. Sin sandbox, el código puede acceder a archivos del sistema, hacer llamadas de red no autorizadas, consumir recursos excesivos o ejecutar código malicioso. Las opciones de sandbox son contenedores Docker efímeros, microVMs como Firecracker, servicios de ejecución de código en la nube como E2B o Modal, o el intérprete de código integrado de la API de OpenAI. Nunca, bajo ninguna circunstancia, des a un agente una herramienta exec sin sandbox.

# BLOQUE_LIMITES

MARIA: [esceptico] Ahora hablemos de lo que falla. Porque esto suena muy limpio en teoría, pero en producción las cosas se rompen. Empecemos por tool hallucination.

IAGO: Es cuando el modelo invoca una herramienta con argumentos incorrectos, invoca una herramienta que no existe en el set disponible, o genera resultados inventados para una herramienta sin haberla llamado realmente.

MARIA: [analitica] Y las mitigaciones son schemas JSON estrictos, validación de argumentos en el cliente antes de ejecutar la herramienta, y logging de todas las llamadas con sus argumentos para auditoría. Pero hay un problema más sutil: definir demasiadas herramientas. Un agente con 50 herramientas disponibles tiene dificultades para seleccionar la correcta. El modelo puede confundir herramientas similares o elegir la menos adecuada. La regla práctica es cinco a quince herramientas por agente bien definidas. Son más efectivas que cincuenta herramientas ambiguamente definidas. Si necesitas más de quince herramientas, probablemente necesitas más de un agente.

IAGO: O agrupar herramientas por contexto y solo cargar el subset relevante según la tarea.

MARIA: [firme] Otro error: descripciones de herramientas demasiado vagas. Una herramienta llamada fetch_data con la descripción "obtiene datos" es inutilizable. El modelo no sabe cuándo usarla ni cómo. Las descripciones deben incluir qué hace exactamente, cuándo usarla versus cuándo no, qué devuelve, y los casos edge importantes. Si no puedes escribir una descripción clara de cien palabras de tu herramienta, la herramienta está mal diseñada.

IAGO: Y hay una distinción técnica importante: herramientas con estado versus sin estado. Las herramientas sin estado, stateless, producen siempre el mismo resultado con los mismos argumentos: búsqueda web, consulta a base de datos. Son las más simples de usar y de testear.

MARIA: [grave] Las herramientas con estado, stateful, modifican el estado del sistema externo y pueden tener efectos secundarios: enviar un email, crear un registro en la base de datos, publicar un mensaje en Slack. Requieren mayor cuidado. El agente no debe llamarlas repetidamente si ya las ejecutó. Y deben tener mecanismos de rollback para los casos donde la acción fue incorrecta. Si tu agente envió cien emails porque entró en un bucle, no hay función de deshacer que los recupere.

MARIA: Primero: function calling es el mecanismo que permite a un modelo LLM invocar funciones externas de forma estructurada mediante bloques JSON con nombre de función y argumentos válidos.

IAGO: Segundo: parallel tool calling reduce la latencia ejecutando múltiples herramientas independientes simultáneamente en lugar de secuencialmente.

MARIA: Tercero: el sandbox es obligatorio para ejecución de código arbitrario. Nunca des a un agente acceso directo a exec sin aislamiento.

IAGO: Y cuarto: cinco a quince herramientas bien definidas con descripciones claras son más efectivas que cincuenta herramientas ambiguas. Tool hallucination se mitiga con schemas estrictos y validación en cliente.
